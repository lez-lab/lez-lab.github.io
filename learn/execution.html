Execution Model

LEZ supports two execution modes in a single protocol: public execution and privacy-preserving execution. Both run the same RISC-V bytecode on the Risc0 zkVM, but they differ in what is revealed on-chain and what guarantees the correctness of the result.

Public Execution

In public execution, the sequencer runs the program directly and proposes a state diff. Validators re-execute the program to verify the result. Everything is transparent — the program inputs, outputs, and state changes are all visible on-chain.

Public execution is fast and simple. It is appropriate for DeFi operations, token transfers, and any logic where privacy is not required.

Privacy-Preserving Execution

In privacy-preserving execution, the user proves the correct execution of a program using a ZK proof generated by the Risc0 zkVM. The sequencer and validators verify the proof but never see the private inputs.

The flow is:

  1. The user runs the program locally with full inputs including private keys and account data.
  2. The Risc0 prover generates a receipt — a cryptographic proof that the program ran correctly and committed specific outputs to its journal.
  3. The user submits the receipt along with the public outputs (new commitments, nullifiers, encrypted account data).
  4. The sequencer verifies the proof and applies the public effects to state.

The program itself does not change between public and private execution. The same RISC-V binary runs in both cases. Privacy is a property of the execution context, not the program.

What Gets Committed Inside a ZKP

The Risc0 receipt proves two things: the program with a specific image ID (hash of the ELF binary) ran, and it committed specific data to its journal. For LEZ programs, the journal contains a ProgramOutput:

struct ProgramOutput {
    instruction_data: Vec<u32>,          // echoed from inputs
    pre_states:       Vec<AccountWithMetadata>,
    post_states:      Vec<AccountPostState>,
    chained_calls:    Vec<ChainedCall>,
    validity_window:  ValidityWindow,
}

The commitment inside the proof cryptographically ties the output to the specific program that produced it. A different program cannot produce the same valid proof for the same output.

The Visibility Mask

Privacy-preserving transactions use a visibility mask — one byte per account — to declare which accounts are public and which are private.

  • 0 — public account: pre and post states are revealed in the clear
  • 1 — private account with authentication: existing private account, requires nullifier secret key
  • 2 — private account, new: uninitialized account being brought into the private state

The privacy circuit processes each account according to its visibility value, computing nullifiers, commitments, and encrypted account data as needed. Validators see only the public accounts, new commitments, consumed nullifiers, and encrypted ciphertexts.

Chained Execution

Both public and private transactions support multi-step atomic execution through chained calls. A program can emit a list of ChainedCall structs in its output, each specifying a subsequent program to invoke, the accounts to pass, and the instruction data.

Execution is depth-first: if program A chains to B and C, and B chains to D, the execution order is A → B → D → C. All calls are atomic — if any step fails, the entire transaction is rolled back.

The flash loan pattern relies on this: a lending program can optimistically transfer funds, invoke a user callback, and then verify repayment — all in a single atomic transaction.

Why This Matters

Most ZK blockchains implement privacy at the application layer. Developers write special private programs and the privacy logic is their responsibility. In LEZ, privacy is a protocol primitive. The same program that handles public token transfers also handles private ones. Developers write programs once; the protocol handles the privacy.

This trades some flexibility for a strong guarantee: if your program runs correctly in public mode, it runs correctly in private mode too.
