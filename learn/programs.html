Writing Programs

A LEZ program is a RISC-V binary that runs inside the Risc0 zkVM. It receives accounts and instruction data as input, performs arbitrary computation, and returns updated account states along with optional calls to other programs.

The Program Model

Programs in LEZ are stateless black boxes. They receive accounts as input, modify them according to their logic, and return the result. They have no direct access to global state unless it is passed as an input account.

The protocol enforces these invariants on every program output:

  • Account IDs are unique in the input set
  • The number of pre-states and post-states must match
  • Account nonces must remain unchanged
  • Program ownership of accounts cannot change (except via the claim mechanism)
  • Only the owning program can decrease an account's balance or modify its data
  • Total balance must be conserved across all accounts in the call

If a program violates any of these rules, the transaction is rejected. This applies identically to public and private execution.

Building with lez-framework

The lez-framework crate provides a #[lez_program] macro that handles the boilerplate, letting you focus on program logic:

use lez_framework::lez_program;

#[lez_program]
fn my_program(accounts: Vec<Account>, instruction: MyInstruction) -> Vec<Account> {
    // your logic here
    accounts
}

The macro handles serialization, the Risc0 guest environment, and ProgramOutput construction. Repository: logos-co/lez-framework

SPEL

SPEL (Smart Program Execution Language) is a higher-level framework for writing LEZ programs. It provides a more ergonomic interface with built-in support for account management, cross-program calls, and testing.

Repository: logos-co/spel · Documentation: spel-agent-resources

Cross-Program Calls

Programs can call other programs atomically by including ChainedCall structs in their output:

ChainedCall {
    program_id:       TOKEN_PROGRAM_ID,
    pre_states:       vec![user_account, vault_account],
    instruction_data: encode(&Transfer { amount: 100 }),
    pda_seeds:        vec![],
}

Calls execute depth-first. All calls in the chain are atomic — a failure at any step rolls back the entire transaction. This is the mechanism behind flash loans and any multi-step DeFi operation.

Program-Derived Accounts (PDAs)

Programs can own accounts whose IDs are derived from the program ID and a seed value. This is how programs hold shared state — liquidity pools, vaults, registries — without requiring a user signature.

When a program emits a chained call with PDA seeds, the execution layer grants authorization for those derived accounts automatically.

Account Ownership and the Claim Mechanism

Every account has a program_owner field. Only the owning program can decrease the account's balance or modify its data.

A program can claim any account that currently has the default owner ([0; 8]) by setting the claim flag in its output. After claiming, the account's program_owner is set to the claiming program's ID. This is how programs create new accounts they control.

Built-In Programs

Token Program

Manages fungible tokens and NFTs. Supports: initialize, mint, burn, transfer, print_nft. The token program is privacy-agnostic — it handles both public and private token accounts identically.

AMM Program

Constant-product AMM with liquidity pools. Supports: new_definition, add_liquidity, remove_liquidity, swap. The AMM chains to the token program for all balance transfers, emitting 2 token calls per swap.

Authenticated Transfer Program

Direct account-to-account transfers with ECDSA signature verification. Used in tutorials and as a reference implementation.
