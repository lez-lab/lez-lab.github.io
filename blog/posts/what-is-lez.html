What is the Logos Execution Zone?

March 15, 2026 ~12 min read

Most blockchains are transparent by default. Every transaction, every balance, every contract interaction is visible to anyone who cares to look. This is fine for some use cases and catastrophic for others. The standard response has been to layer privacy on top: mixers, shielded pools, privacy-focused dApps wrapping public chains. These work, but they are bolted on. The underlying execution model was never designed for confidentiality.

The Logos Execution Zone takes a different approach. LEZ is a Layer 2 blockchain where privacy is a first-class primitive of the protocol itself. Programs deployed to LEZ can natively operate over both public and private state without relying on external privacy tooling. This is not an optional feature or a special mode. It is how the system works.

The Problem LEZ Solves

Consider a simple scenario: a payroll program on a public blockchain. Even if you encrypt the amounts, the transaction graph reveals who pays whom and when. Observers can infer salary structures, contractor relationships, and organizational hierarchies. Wrapping this in a privacy layer helps, but you are fighting the architecture rather than working with it.

LEZ addresses this at the protocol level. A payroll program on LEZ can store employee records as private state. The program can verify that disbursements are correct, that totals balance, and that authorization rules are followed, all without revealing the underlying data to anyone except the parties involved. The chain verifies the computation happened correctly. It never sees the inputs.

This matters beyond payroll. Voting, medical records, supply chain logistics, identity verification, sealed-bid auctions, compliance reporting. Any domain where "the computation must be verifiable but the data must remain confidential" is a natural fit for LEZ.

Hybrid Public/Private State

The core design decision in LEZ is the hybrid state model. Every program on LEZ has access to two kinds of state.

Public state works the way you expect from other blockchains. It is stored on-chain in the clear. Anyone running a node can read it. It is useful for things that should be transparent: governance proposals, token metadata, protocol parameters, public registries.

Private state is different. The actual data is never written to the chain. Instead, the chain stores cryptographic commitments, which are hashes that attest to the existence and integrity of data without revealing it. When a program needs to update private state, the user executes the computation locally and generates a zero-knowledge proof demonstrating that the update is valid according to the program's rules. The chain verifies the proof and records the new commitment. The data itself stays with the user.

These two state modes coexist within a single program. A token program might store the total supply publicly (so anyone can audit it) while keeping individual balances private (so only the holder knows their balance). A governance program might publish proposal text publicly while keeping individual votes private until the voting period ends.

The hybrid model means developers do not have to choose between a "privacy chain" and a "normal chain." They choose, per field, per data structure, what should be visible and what should be confidential.

How Private Execution Works

Private execution in LEZ is built on Risc0, a zero-knowledge virtual machine that runs the RISC-V instruction set. This is a significant architectural choice. RISC-V is a real, general-purpose CPU instruction set. It is not a custom DSL or a constrained circuit language. Programs compiled to RISC-V can express arbitrary computation, which means LEZ programs are not limited in what logic they can execute privately.

The flow works like this:

  1. A user wants to execute a private operation, say, transferring a private token balance.
  2. They run the program logic locally on their own machine. This includes reading their private state, computing the updated balances, and producing the new state.
  3. Risc0 generates a zero-knowledge proof (ZKP) that the computation was performed correctly according to the program's code. The proof attests: "this program was executed with some inputs, and it produced these outputs, and the execution followed all the rules." It does not reveal what the inputs were.
  4. The user submits the proof and the new state commitments to the chain.
  5. The chain verifies the proof, checks that the old commitments are valid and unspent, and records the new commitments.

This is a meaningful departure from how most smart contract platforms work. On Ethereum, a contract's logic runs on every validator. On LEZ, private logic runs on the user's machine, and the chain only verifies the proof. This has implications for scalability (the chain does less work per transaction) and privacy (the chain never touches the raw data).

The Account Model

LEZ uses a UTXO-inspired account model designed for privacy. The key primitives are:

AccountId is the public identifier for an account. It is derived from the account's public key and is used to receive commitments.

nsk (nullifier secret key) is a private key that the account holder uses to spend commitments. It is never shared. When you spend a commitment, you produce a nullifier derived from the nsk and the commitment. The nullifier proves you are authorized to spend without revealing which commitment you are spending.

CommitmentSet is the on-chain set of all active commitments. When private state is created or updated, a new commitment is added to this set. Commitments are hashes. They reveal nothing about the underlying data.

NullifierSet is the on-chain set of all nullifiers that have been used. When a commitment is spent, its nullifier is added to this set. This prevents double-spending: you cannot spend the same commitment twice because the nullifier would already exist in the set.

The interaction between commitments and nullifiers is what makes private state transfers possible without revealing the connection between sender and receiver. A transfer consumes old commitments (by publishing their nullifiers) and creates new commitments (by adding hashes to the commitment set). An observer sees nullifiers being added and commitments being added, but cannot link them without the private keys.

Writing Programs

LEZ programs are written in Rust using the lez-framework crate. The framework provides the #[lez_program] attribute macro, which handles the boilerplate of state management, proof generation interfaces, and chain interaction.

A minimal program looks something like this:

use lez_framework::prelude::*;

#[lez_program]
mod my_program {
    #[state]
    pub struct Counter {
        #[public]
        pub count: u64,
    }

    #[action]
    pub fn increment(state: &mut Counter) {
        state.count += 1;
    }
}

The #[state] attribute defines the program's state structure. Fields marked #[public] are stored on-chain in the clear. Fields without this annotation are treated as private state, committed as hashes.

The #[action] attribute defines operations that modify state. For public state, these execute on-chain like traditional smart contracts. For private state, the framework generates the Risc0 guest code needed to prove the execution locally.

Programs are compiled to RISC-V (for private execution in Risc0) and to the chain's native format (for public execution and verification). The lez-framework handles this dual compilation transparently.

SPEL

SPEL (Simple Program Execution Language) is the transaction format for interacting with deployed programs. It is a compact, human-readable language for describing program calls, batched operations, and conditional execution. Think of it as LEZ's equivalent of Ethereum's calldata, but designed to be readable and composable.

The spelbook repository collects SPEL recipes for common patterns: token transfers, multi-sig operations, governance votes, and more. It is a practical reference for developers learning how to interact with LEZ programs.

Where to Go Next

If you want to start building on LEZ, here is a reasonable path:

  1. Read through the cryptolizards tutorial. It walks you through building a private NFT collection from scratch, covering state design, private minting, and proof generation. It is the most complete end-to-end example available.
  2. Study the lez-framework documentation and examples. The framework's test suite is particularly instructive for understanding the state model.
  3. Browse the spelbook for SPEL patterns. It will give you a feel for how programs are called in practice.
  4. Check the spel-agent-resources repository for tooling around programmatic SPEL generation and agent-driven program interaction.
  5. Follow LEZ Lab for ecosystem updates, new guides, and curated resources as the ecosystem develops.

LEZ is early. The protocol is under active development, the tooling is maturing, and the community is small but engaged. That also means this is the stage where understanding the fundamentals pays off the most. The architecture is sound, the design space is wide open, and there is meaningful work to be done at every level of the stack.
