How Private Execution Works in LEZ

March 20, 2026 ~15 min read

The first post in this series introduced LEZ's hybrid public/private state model and explained the high-level flow of a private transaction. This post goes deeper. We will walk through the actual mechanics: how private accounts are represented, what happens inside the privacy-preserving circuit, what the visibility mask controls, and what a privacy-preserving transaction looks like at the data structure level. The goal is to give you enough understanding to reason about private programs, not just use them.

This is not about application-level encryption. LEZ does not encrypt your data and store the ciphertext on-chain. Private execution in LEZ means that the computation itself happens off-chain, inside a zero-knowledge proof. The chain never sees the private inputs, never executes the private logic, and never stores the private state. It only verifies that the computation was performed correctly according to the program's rules.

The Commitment/Nullifier Model

Every account in LEZ, whether public or private, shares the same schema:

struct Account {
    program_owner: ProgramId,
    balance: u128,
    data: Data,
    nonce: u128,
}

For public accounts, this data lives in the clear on-chain. For private accounts, the chain stores a cryptographic commitment instead. A commitment is a SHA-256 hash that binds the account's contents to its owner without revealing either:

Commitment = SHA256(
    domain_separator ||
    nullifier_public_key ||
    program_owner ||
    balance ||
    nonce ||
    SHA256(data)
)

The commitment is unconditionally hiding. Even if you know the program owner and suspect a particular balance, you cannot confirm it because the nonce acts as a blinding factor. For private accounts, the nonce is not a sequential counter like it is for public accounts. It is a pseudorandom value derived iteratively from the nullifier secret key: nonce_i = SHA256(nsk || nonce_{i-1}). This ensures each commitment is unique even if the account data has not changed.

Commitments are stored in a Merkle tree called the CommitmentSet. When a new private account is created or an existing one is updated, a new commitment is inserted into the tree. The chain maintains the full history of Merkle roots, which matters for proof verification.

To spend or update a private account, you must produce a nullifier. A nullifier is a value that proves you are authorized to consume a particular commitment, without revealing which commitment you are consuming. There are two cases:

When initializing a new private account:

Nullifier = SHA256("/LSSA/v0.3/Nullifier/Initialize/" || npk)

When updating an existing account:

Nullifier = SHA256("/LSSA/v0.3/Nullifier/Update/" || commitment || nsk)

The nullifier for an update requires the nullifier secret key (nsk), which only the account owner possesses. The chain maintains a NullifierSet of all nullifiers that have been published. If a nullifier already exists in the set, the transaction is rejected. This is how double-spending is prevented: you cannot consume the same commitment twice because the nullifier would collide.

The key insight is that commitments and nullifiers are unlinkable to outside observers. When a transaction publishes nullifiers and new commitments, an observer sees that some old state was consumed and some new state was created, but cannot determine which commitments the nullifiers correspond to. The connection between inputs and outputs is known only to the parties who hold the private keys.

Private Account Keys

A private account in LEZ is identified by three key components:

The nullifier secret key (nsk) is the account's spending authority. It is a 32-byte secret used to derive nullifiers. The nsk is never shared and never leaves the owner's machine. Knowledge of nsk is what constitutes "ownership" of a private account.

The nullifier public key (npk) is derived from the nsk and serves as the account's identity. The AccountId for a private account is computed as SHA256("/LSSA/v0.3/AccountId/Private/" || npk). This is the identifier used within program logic.

The incoming viewing key pair (isk/ipk) enables the account owner to detect and decrypt incoming transactions. When someone sends you a private transfer, they encrypt the new account state using a Diffie-Hellman shared secret derived from their ephemeral key and your incoming viewing public key (ipk). Only the holder of the incoming viewing secret key (isk) can derive the same shared secret and decrypt the data.

This separation of concerns matters. The nsk controls spending. The isk controls viewing. A watch-only wallet could hold the isk to track balances without having the ability to spend.

What a Privacy-Preserving Transaction Looks Like

A privacy-preserving transaction in LEZ has two components: a message and a witness set.

The message is the public-facing part of the transaction. It contains:

Message {
    public_account_ids: Vec<AccountId>,
    nonces: Vec<Nonce>,
    public_post_states: Vec<Account>,
    encrypted_private_post_states: Vec<EncryptedAccountData>,
    new_commitments: Vec<Commitment>,
    new_nullifiers: Vec<(Nullifier, CommitmentSetDigest)>,
}

The public_account_ids and public_post_states describe any changes to public accounts. In a shielding transaction (public to private), you would see a public account's balance decrease. In a deshielding transaction (private to public), a public account's balance would increase. For a fully private transfer, these lists may be empty.

The encrypted_private_post_states contain the encrypted account data for each new private output. Each entry includes the ciphertext, an ephemeral public key (for Diffie-Hellman), and a view tag. The view tag is a single byte derived from the recipient's keys that allows wallet software to quickly filter transactions without attempting full decryption on every one.

The new_commitments are the hashes that will be inserted into the CommitmentSet. The new_nullifiers are the values that will be inserted into the NullifierSet, each paired with the CommitmentSet digest (Merkle root) that the nullifier's membership proof was computed against.

The witness set contains a ZK proof and any signatures needed for public account authorization:

WitnessSet {
    proof: Proof,
    signatures_and_public_keys: Vec<(Signature, PublicKey)>,
}

The proof is a Risc0 ZKP that attests to the correctness of the entire transaction. The signatures authorize any public accounts involved. Purely private transactions do not need signatures because the ZK proof itself serves as authorization, since only the holder of the nsk could produce valid nullifiers.

Inside the Privacy-Preserving Circuit

The privacy-preserving circuit is the core of LEZ's private execution model. It is a Risc0 guest program that runs inside the zkVM and produces a proof that the sequencer can verify. This circuit is not a "program" in the LEZ sense. Programs define application logic (transfers, token operations, governance). The privacy circuit is infrastructure: it wraps program execution, handles the public/private account split, computes commitments and nullifiers, and encrypts outputs.

The circuit receives the following inputs:

PrivacyPreservingCircuitInput {
    program_outputs: Vec<ProgramOutput>,
    visibility_mask: Vec<u8>,
    private_account_keys: Vec<(NullifierPublicKey, SharedSecretKey)>,
    private_account_nsks: Vec<NullifierSecretKey>,
    private_account_membership_proofs: Vec<Option<MembershipProof>>,
    program_id: ProgramId,
    deposit_creates: Vec<DepositCreateInput>,
    deposit_claims: Vec<DepositClaimInput>,
}

The first thing the circuit does is verify program execution. It takes the program_outputs and calls env::verify() for each one. This is Risc0's assumption verification: the program was already executed and proved separately, and the circuit verifies those proofs compose correctly. It also runs validate_execution() to check that the program behaved according to LEZ's rules (balances conserved, data only modified by the owning program, nonces unchanged, and so on).

If the program uses chained calls (one program invoking another), the circuit processes the full call chain, verifying each step and ensuring that pre-states and post-states are consistent across calls.

The Visibility Mask

After verifying program execution, the circuit splits the accounts into public and private outputs using the visibility mask. This is a byte array with one entry per account. The values are:

  • 0 = public account. Pre-state and post-state are included in the circuit output in the clear.
  • 1 = private account with authentication. The account owner provided their nsk, proving they are authorized to spend. This is used for the sender's account in a private transfer.
  • 2 = private account without authentication. This is a new account being initialized for the first time. No nsk is needed because there is no prior commitment to nullify.

The visibility mask is what makes LEZ's hybrid model work at the transaction level. A single program execution can involve both public and private accounts, and the mask determines how each account's state transition is handled.

Processing Private Accounts

For each private account (visibility mask 1 or 2), the circuit performs several operations:

For authenticated accounts (mask = 1), the circuit verifies the nullifier public key matches the AccountId, checks the nsk against the npk, verifies the membership proof against the CommitmentSet, and computes both the nullifier and the new commitment set digest. The nonce is incremented using SHA256(nsk || previous_nonce).

For new accounts (mask = 2), the circuit asserts the pre-state is the default account (all zeros), computes an initialization nullifier from the npk, and sets the initial nonce using SHA256(npk). No membership proof is required because there is no prior commitment in the tree.

In both cases, the circuit then:

  1. Computes the new commitment: Commitment::new(npk, &post_state_with_updated_nonce)
  2. Encrypts the post-state using the shared secret derived from the sender's ephemeral key and the recipient's viewing public key
  3. Adds the commitment and ciphertext to the circuit output

The encryption happens inside the circuit. This is a deliberate design choice. By encrypting within the ZKP, the protocol can prove that the ciphertext corresponds to the committed plaintext. A malicious sender cannot publish a fake ciphertext that the recipient cannot decrypt, because the proof would not verify.

What the Circuit Commits

The circuit's output, which becomes the public journal of the ZK proof, contains:

PrivacyPreservingCircuitOutput {
    public_pre_states: Vec<AccountWithMetadata>,
    public_post_states: Vec<Account>,
    ciphertexts: Vec<Ciphertext>,
    new_commitments: Vec<Commitment>,
    new_nullifiers: Vec<(Nullifier, CommitmentSetDigest)>,
    new_deposit_commitments: Vec<DepositCommitment>,
    encrypted_deposit_notes: Vec<Ciphertext>,
    deposit_nullifiers: Vec<(DepositNullifier, CommitmentSetDigest)>,
}

This output is what the sequencer verifies against. The proof attests that all the values in this output were correctly derived from valid program execution, valid key material, and valid membership proofs, without revealing what the private inputs were.

How the Sequencer Validates a Privacy-Preserving Transaction

When the sequencer receives a privacy-preserving transaction, it performs the following checks in order:

  1. The transaction must have at least one commitment or nullifier (account or deposit level). Empty transactions are rejected.
  2. No duplicate account IDs, nullifiers, or commitments within the transaction.
  3. Nonce checks: each signer's nonce must match the current on-chain nonce for their public account.
  4. Signature verification: all signatures must be valid BIP-340 Schnorr signatures over the message.
  5. Proof verification: the ZK proof is verified against the circuit output reconstructed from the message fields. This is the core check. If the proof verifies, the sequencer knows that the program was executed correctly, balances are conserved, nullifiers are valid, and commitments are properly derived.
  6. Commitment freshness: none of the new commitments already exist in the CommitmentSet.
  7. Nullifier uniqueness: none of the new nullifiers already exist in the NullifierSet, and each nullifier's CommitmentSetDigest appears in the root history.

If all checks pass, the sequencer applies the state transition: public account states are updated, new commitments are inserted into the Merkle tree, and nullifiers are added to the NullifierSet.

The Hybrid Model in Practice

The power of LEZ's approach is that programs do not need to be written differently for public and private execution. The same program logic, the same Rust code, handles both. The visibility mask is external to the program. The program receives accounts, processes them according to its rules, and outputs updated accounts. The privacy circuit then decides, based on the mask, which outputs are published in the clear and which are committed and encrypted.

Consider a simple token transfer. The authorized transfer program takes a sender and recipient, checks authorization, debits the sender, and credits the recipient. This program is identical whether the transfer is public-to-public, public-to-private (shielding), private-to-public (deshielding), or fully private.

For a public transfer, the visibility mask is [0, 0]. Both accounts are public. The transaction is a standard public transaction and does not go through the privacy circuit at all.

For a shielding transfer (public sender, private recipient), the visibility mask is [0, 2]. The sender's state change is published in the clear, and the recipient's state is committed and encrypted. The observer sees that the sender's balance decreased but cannot determine where the funds went.

For a fully private transfer, the visibility mask is [1, 2] (existing private sender, new private recipient) or [1, 1] (both existing private accounts). The observer sees nullifiers and commitments but cannot link them or determine the amounts involved.

How This Differs from Aztec

Aztec is the closest comparable system, so the comparison is worth making explicit.

Aztec uses a UTXO-based note model where private state is stored as encrypted notes. Each note is an independent piece of state that is created and destroyed. LEZ uses an account model where private accounts have the same structure as public accounts (program owner, balance, data, nonce) and are updated in place, with commitments replacing old state.

In Aztec, developers write contracts in Noir (a domain-specific ZK language) and must explicitly manage the distinction between public and private functions. Public functions run on the sequencer; private functions run on the client. Cross-function calls between public and private contexts are asynchronous and require careful handling. In LEZ, the program logic is written once in Rust, compiled to RISC-V for Risc0, and the public/private split is handled by the visibility mask at transaction construction time, not at the language level.

Aztec's private execution produces a "kernel proof" that is built up recursively as the transaction moves through its lifecycle. LEZ takes a different approach: the program is executed and proved first, and then the privacy-preserving circuit wraps the program proof along with all the commitment/nullifier/encryption logic into a single composite proof. The Risc0 assumption mechanism makes this composition straightforward.

The trade-off is generality versus specialization. Aztec's Noir circuits are highly optimized for the specific computations they prove. LEZ's Risc0-based approach lets you write arbitrary Rust but pays for it in proof generation time, since RISC-V execution inside a zkVM is inherently less efficient than hand-optimized circuits. For LEZ, the bet is that general-purpose programmability and developer ergonomics matter more than proof generation speed, especially as Risc0's proving infrastructure improves.

What Developers Need to Understand

If you are writing programs that will be used in private transactions, there are several things to keep in mind.

Programs are visibility-agnostic. Your program logic should not need to know whether an account is public or private. The same code handles both. The visibility mask is applied externally by the transaction builder.

Authorization works differently for private accounts. For public accounts, authorization comes from a BIP-340 Schnorr signature. For private accounts, authorization is implicit: the ability to produce a valid nullifier (which requires the nsk) constitutes proof of ownership. Your program checks is_authorized on the AccountWithMetadata, and the infrastructure handles the rest.

Balance conservation is enforced by the protocol. The validate_execution function checks that the sum of all account balances before execution equals the sum after. You cannot create or destroy tokens. This check operates on u256 arithmetic to prevent overflow exploits on the u128 balance field.

Programs cannot change program ownership or nonces. These fields are managed by the protocol. The nonce for private accounts is updated by the privacy circuit (using the pseudorandom derivation from nsk), not by the program.

Chained calls work in privacy transactions but with a constraint. The current privacy circuit supports a linear chain of tail-calls. A program can invoke another program, which can invoke another, forming a chain. However, a single program cannot invoke multiple other programs in parallel within the same execution. This means certain patterns like the AMM swap (which makes three separate token program calls) are not yet supported in the privacy path.

Account scanning is a client-side operation. When you receive a private transfer, your wallet scans new transactions by checking the view tag (a single byte derived from your keys) against each encrypted output. If the view tag matches, the wallet attempts decryption using the Diffie-Hellman shared secret. This is efficient because the view tag filters out roughly 255 of every 256 transactions before any expensive decryption is attempted.

Proof generation happens on the client. The user's machine executes the program, generates the Risc0 proof, and submits the complete transaction with proof to the sequencer. This has implications for user experience: proof generation takes time and computational resources. The sequencer's job is only to verify, which is fast.

The Deposit System

LEZ also supports a deposit mechanism that allows transfers between private accounts without requiring the sender to know the recipient's full account state. Deposits work through their own commitment/nullifier pair (DepositCommitment and DepositNullifier), separate from the account commitments. The sender creates a deposit note with the transfer amount, commits it, and encrypts it for the recipient. The recipient later claims the deposit by proving knowledge of the note and producing a deposit nullifier. This two-phase approach decouples the send and receive steps, which is useful when the sender does not have access to the recipient's current commitment in the Merkle tree.

Looking Forward

The privacy-preserving execution model in LEZ is functional and well-specified, but it is also under active development. The linear chained-call constraint will likely be relaxed. Proof generation times will improve as Risc0's prover matures. The deposit system is new and still being refined.

What is already solid is the architecture. The commitment/nullifier model, the visibility mask approach, the separation of program logic from privacy infrastructure, and the in-circuit encryption guarantee are all design choices that will hold as the system scales. Understanding these fundamentals now will serve you well as the protocol evolves.
