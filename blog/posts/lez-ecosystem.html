The LEZ Ecosystem — What is Being Built

March 18, 2026 ~10 min read

The Logos Execution Zone is still early, but it is not empty. There is a growing set of repositories, tools, programs, and people working across the stack. This post maps what exists as of early 2026: what each piece does, where it lives, and how the parts connect.

This is a field report, not a marketing document. Some of these projects are production-ready. Others are prototypes or works in progress. The goal is to give you an honest picture of the landscape so you can orient yourself and find where to contribute.

Core Protocol

The foundation of LEZ lives under the logos-blockchain GitHub organization. The main repository, logos-execution-zone, contains the node implementation, consensus logic, the hybrid state engine, and the proof verification layer. This is where the protocol rules are defined: how commitments are stored, how nullifiers are checked, how public and private state transitions are validated.

The core protocol is implemented in Rust and is structured as a set of crates that can be used independently. The state engine, for example, can be embedded in tooling without pulling in the full node. The proof verifier is a standalone crate that other projects use to validate ZKPs without running a node.

Development on the core protocol moves steadily. The team ships regularly, and breaking changes are communicated through release notes and migration guides. If you are building on LEZ, watching the core repo's releases is worth your time.

Official Programs

The lez-programs repository contains programs maintained by the core team. These are the canonical implementations of common primitives: fungible tokens, non-fungible tokens, governance modules, and access control patterns.

These programs serve two purposes. They provide ready-to-use functionality for developers who need standard building blocks. They also serve as reference implementations that demonstrate how to use the framework correctly. If you are unsure how to handle a particular pattern (say, combining public metadata with private ownership), the official programs are the first place to look.

The token program is the most mature. It supports both public and private balances, transfer with proof generation, and a clean SPEL interface for wallet integration. The governance program is newer and actively evolving.

Framework and Tooling

lez-framework

The lez-framework crate is the primary SDK for writing LEZ programs. It provides the #[lez_program] macro, state management primitives, and the dual compilation pipeline that targets both on-chain execution and Risc0 guest programs. Most developers interact with LEZ through this crate.

The framework handles a significant amount of complexity behind its macros. State serialization, commitment generation, nullifier derivation, proof circuit setup. Developers write what looks like normal Rust, and the framework generates the cryptographic machinery underneath.

SPEL

SPEL (Simple Program Execution Language) is the transaction description format for LEZ. It is designed to be human-readable, composable, and suitable for both manual construction and programmatic generation. A SPEL transaction describes which programs to call, with what parameters, and in what order.

SPEL is important because it is the interface between users (and their wallets) and on-chain programs. A well-designed SPEL interface makes a program easy to integrate. The format supports batching (multiple calls in one transaction), conditionals (execute step B only if step A succeeds), and references (pass the output of one call as input to another).

logos-scaffold

logos-scaffold is a project generator in the style of cargo-generate. It creates a new LEZ program project with the correct directory structure, build configuration, and test setup. It is the fastest way to go from zero to a compiling program.

lezard

lezard is the CLI companion for LEZ development. It wraps common workflows: compiling programs, running local test networks, deploying to testnets, generating and verifying proofs, and interacting with deployed programs via SPEL. If you are developing on LEZ, lezard is your day-to-day tool.

The name is, obviously, a lizard pun. The LEZ ecosystem leans into the reptile theme. You will encounter it frequently.

Community Programs

Beyond the official programs, a growing set of community-built projects explores what LEZ can do. These vary in maturity, but they represent the creative frontier of the ecosystem.

lez-multisig

A multi-signature wallet program for LEZ. It supports configurable threshold signing (M-of-N), timelocks, and private signer sets. The private signer feature is particularly interesting: the set of signers is committed as private state, so observers cannot see who controls a multisig. Only valid signatures (proven via ZKP) can authorize transactions.

payment-streams

A streaming payments program. It allows continuous, per-second payment flows between accounts. Useful for salaries, subscriptions, and service fees. The implementation uses private balances, so the payment rate and accumulated amounts are confidential between sender and receiver.

eth-lez-atomic-swaps

A cross-chain atomic swap protocol between Ethereum and LEZ. It uses hash time-locked contracts (HTLCs) on both sides to enable trustless exchange of assets. This is one of the first bridges between LEZ and another chain, and it demonstrates how LEZ's commitment model interacts with external systems.

spelbook

A curated collection of SPEL recipes. Think of it as a cookbook for LEZ program interaction. It contains patterns for common operations: token transfers, batch minting, governance proposals, multi-step workflows. Each recipe includes the SPEL code, an explanation of what it does, and notes on when to use it.

spelbook is one of the most useful resources for new developers because it shows real patterns in context rather than abstract documentation.

cryptolizards

A tutorial project that walks through building a private NFT collection on LEZ. It is the most complete beginner-to-intermediate guide available. The tutorial covers state design, private minting, ownership proofs, transfer logic, and SPEL integration. It is structured as a series of steps, each building on the last, with working code at every stage.

cryptolizards is where most new LEZ developers start, and it is the project that LEZ Lab recommends first.

music-escrow

An escrow program designed for music licensing. Artists deposit tracks (represented as on-chain metadata with off-chain storage references), and buyers can purchase licenses through an escrow flow that protects both parties. The private state model is used to keep licensing terms and payment details confidential.

rln (Rate Limiting Nullifier)

An implementation of the Rate Limiting Nullifier protocol on LEZ. RLN allows anonymous participation in systems (messaging, voting, API access) while enforcing rate limits. If a participant exceeds the limit, their identity can be revealed. On LEZ, this is implemented using the native commitment and nullifier primitives, which makes it more efficient than standalone RLN implementations.

Wallet and Module Layer

The wallet layer is where users interact with LEZ. Wallets need to handle both public and private state, which means managing commitment trees, running local proof generation, and presenting a coherent view of balances that may be partially private.

The current wallet implementations are mostly developer-focused. There is no polished end-user wallet yet, but the building blocks exist: key management, commitment tracking, proof generation, and SPEL transaction construction. The lezard CLI can serve as a functional wallet for development and testing.

The module layer refers to the pattern of composing programs. LEZ programs can call other programs, which enables modular architectures. A DeFi protocol might compose the token program, the multisig program, and a custom lending program into a single workflow. The SPEL batching format makes this composition explicit and auditable.

The Shape of Things

Looking at the ecosystem as a whole, a few things stand out.

The core is solid. The protocol, framework, and tooling form a coherent stack that developers can actually use. The dual public/private state model is not just a whitepaper concept. It is implemented, tested, and used by real programs.

The community layer is early but real. The community programs range from practical (multisig, payment streams) to exploratory (music escrow, RLN). They demonstrate that the framework is expressive enough for diverse use cases.

The biggest gap is in the user-facing layer. Wallets, block explorers, and onboarding flows are still developer-grade. This is normal for a project at this stage, but it is also where contributions would have outsized impact.

If you are looking to get involved, the ecosystem is small enough that meaningful contributions are noticed and the community is welcoming. Start with LEZ Lab for curated guides, try the cryptolizards tutorial to get your hands dirty, and explore the spelbook to understand the patterns. The foundation is here. What gets built on it is up to the people who show up.
